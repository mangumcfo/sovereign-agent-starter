# Sovereign Stack Audit — 2026-06-13d (W7 confirming sweep)

Seven-dimension review (security, performance, code quality, test coverage, dependencies, architecture, constitutional conformance) with adversarial verification — every finding below survived an attempt to refute it; one candidate was refuted and is recorded for the record.

## Executive Summary

**Overall health: 82/100.** The engine core is mature and well-defended: the hash-chained obligation ledger has a proven cross-process write-fence, memoized hot paths, evidence tiers and resolve-at-entry validation; prior security audits closed the classic issues (principal spoofing, shell injection, path traversal, wildcard CORS, hardcoded secrets) and 206 tests pass in ~8s. Points come off for one **CRITICAL** false-success-on-the-immutable-chain bug in the EXECUTE half, a token-less owner-auth bypass via DNS rebinding, a portability regression that breaks doc-serving off the operator's host, and a cluster of structural/coverage/consistency gaps. **Critical count: 1.** The single most important theme: **the hardening discipline applied to the ledger core (loud failure, one-resolver, fail-closed) has not yet reached the EXECUTE half — atrium_apply/atrium_executor commit, seal, and close with weaker guards, less test coverage, and machine-bound paths than the chain they write to.**

## Prioritized Findings

### CRITICAL

**1. Constitutional ERROR VOICE / TRUTH · `scripts/atrium_apply.py:282-286` (seal/close 287-316)**
`git commit` result is captured into `c` but `c.returncode` is never checked; `_git` never raises. After commit the code unconditionally runs `git rev-parse --short HEAD` and records that hash. On a failed commit (pre-commit hook rejection, no-op commit, identity error, surviving index.lock) `rev-parse` returns the PRIOR commit, and the script seals a cylinder and closes the obligation with E2 evidence `"...auto-applied (KM accepted) — commits {commits}; sealed; tests green."` — a false success on the hash-chained immutable ledger citing a commit that does not contain the changes. The adjacent close step already hard-fails for exactly this class (H1 guard, lines 295-300); the commit step has no equivalent.
*Fix:* after line 283, if `c.returncode != 0` (excluding benign nothing-to-commit) call `_revert(changed, created)` + `_mark_error(pid, 'git commit failed: ' + (c.stderr or c.stdout))` and return non-zero BEFORE sealing/closing; capture the hash only on a verified commit (compare HEAD before/after, or parse commit stdout). *Effort:* 20-30 min (mirrors existing `_mark_error`/`_revert`). *Verifier:* CONFIRMED — `c` is dead after assignment; tests run before commit so they don't guard it; a node-attested E2 receipt can assert a non-existent commit.

### HIGH

**2. Auth / DNS-rebinding bypass of loopback-trust · `src/sovereign_agent/node_api/auth.py:102-104, 110-121, 157-175`**
The loopback-owner shortcut grants full node-owner authority (passes both `require_principal` and `require_owner`) to any token-less request whose `remote_addr` is loopback and whose `Sec-Fetch-Site` is not cross-site/same-site. There is no Host-header validation anywhere in src. An attacker page on `http://evil.example:8421` that rebinds the domain to 127.0.0.1 makes `remote_addr` loopback while the browser stamps `Sec-Fetch-Site: same-origin` (which `_is_cross_site_browser_write` explicitly allows), so the CSRF guard never fires. Result: owner-only code-executing routes (POST `/produce`, `/proposals/<id>/apply`, `/recompile`, `/feedback/<id>/disposition`, `/proposals/<id>/decide`) fire as the owner and mutate the chain. CORS does not mitigate (governs response readability, not whether the POST executes).
*Fix:* add a Host-header allowlist gate to the loopback shortcut (ideally a `before_request`), rejecting any Host not in `{127.0.0.1[:port], localhost[:port], [::1][:port]}` before honoring `_loopback_owner()`; add a security test asserting `Host: evil.example` is refused. *Effort:* 30-45 min. *Verifier:* CONFIRMED, could not refute; no SERVER_NAME/HTTP_HOST/trusted-hosts check exists; the only `before_request` just short-circuits OPTIONS. Caveat: requires opt-in `BREATHLINE_NODE_LOOPBACK_OWNER` (the documented personal-node posture) and no browser/OS PNA protection.

`_doc_roots()` hardcodes `_P("/home/kmangum/work-repos/mangumcfo/breathline-books-vault")` (used as `vault / "kdp"`) for the /doc route's allowed roots, while every other vault consumer routes through `config.get_books_kdp_root()`. On KM's host today the literal exists and the env is unset, so it functions locally; the cross-surface divergence (other surfaces honor `BREATHLINE_BOOKS_VAULT`, /doc stays pinned) and any non-KM host are host-independent breakages.
